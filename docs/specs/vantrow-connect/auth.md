## Model

Authentication in v1 is **per-tenant scoped API keys**:

- The **producing platform issues** keys, one tenant per key. A key can never
  read another tenant's data — tenancy is bound into the credential, so there
  is no tenant parameter anywhere in the API.
- The **Vantrow dashboard stores** the key (encrypted at rest, server-side
  only — never in browser code, mobile apps, or client-visible config) and
  presents it on every request:

```
Authorization: Bearer vc_live_k7m2p9d4hn7eqa2rv8kfswd5sx9mgt3z
```

- `/v1/health` is the only unauthenticated endpoint.

## Key format

```
vc_<mode>_<random>
```

- `vc_live_...` — production data. `vc_test_...` — sandbox/demo data.
- `<random>` is at least 32 characters of URL-safe entropy (>= 160 bits),
  generated by a CSPRNG.
- Keys are shown **once** at issuance. The platform stores only a hash;
  a lost key is replaced, never recovered.
- The stable `vc_` prefix and mode segment make keys recognizable to secret
  scanners; treat any string matching `vc_(live|test)_[0-9a-zA-Z_-]{32,}` as
  a credential.

## Scopes

Each key carries an explicit scope set, requested at issuance:

| Scope | Grants |
|---|---|
| `read:projects` | `GET /v1/projects`, `GET /v1/projects/{id}` |
| `read:contacts` | `GET /v1/contacts` |
| `read:invoices` | `GET /v1/invoices` |
| `read:metrics` | `GET /v1/metrics` |

- All v1 scopes are read-only by design — there are no write scopes to grant.
  A typical dashboard key carries all four; a metrics-only wallboard key
  carries just `read:metrics`.
- Missing/invalid key → `401 unauthorized`. Valid key without the required
  scope → `403 forbidden`. Both use the standard error envelope
  (`code`, `message`, `request_id`).
- Scopes gate **endpoints**, not fields. Extension data rides on entities and
  is covered by the entity's scope.

## Key rotation (dual-key overlap)

The platform supports **two active keys per tenant** so rotation is
zero-downtime:

1. Issue key B (same tenant, same scopes). Keys A and B are both live.
2. Deploy key B to the dashboard configuration.
3. Verify traffic on key B (the platform exposes last-used-at per key to its
   operators).
4. Revoke key A. Revocation is immediate; subsequent requests with A get
   `401`.

Recommended overlap window: hours, at most **7 days**. Rotate immediately on
suspected exposure, personnel change, or scanner hit; rotate routinely at
least every 12 months.

## Webhook secrets

Webhook endpoints are provisioned with a signing secret, format
`whsec_<random>` (same entropy rules as API keys), delivered out-of-band by
the platform operator in v0. It is used only to verify
`X-Vantrow-Signature` (see `events.md`) — never sent as a bearer token.

**Secret rotation (dual-secret overlap):**

1. Operator adds secret B alongside secret A for the endpoint.
2. While both are active, every delivery is signed with **both** secrets —
   the `X-Vantrow-Signature` header carries one `v1=` entry per active
   secret. Verifiers accept if **any** entry matches, so an endpoint knowing
   only A or only B keeps working throughout.
3. Consumer switches to verifying with B.
4. Operator removes A. Recommended overlap: at most **24 hours**.

## Transport requirements

- **TLS only** (HTTPS, TLS 1.2 minimum; 1.3 preferred) — for both API calls
  and webhook deliveries. Plain-HTTP webhook endpoint URLs are rejected at
  provisioning time; plain-HTTP API requests are refused (no redirect
  upgrade, so credentials are never emitted over cleartext even once).
- API keys travel **only** in the `Authorization` header — never in URLs,
  query strings, or request bodies (URLs leak into logs and referrers).
- Producers should send HSTS; consumers must not disable certificate
  verification.

## Upgrade path: OAuth 2.0 client credentials (NOT in v1)

The documented forward path for machine-to-machine auth is **OAuth 2.0
client-credentials** (RFC 6749 §4.4): per-dashboard `client_id`/`client_secret`
exchanged at a token endpoint for short-lived access tokens carrying the same
scope vocabulary, enabling central revocation, token expiry, and future
`mTLS`/`private_key_jwt` client authentication.

This is **explicitly not part of v1**. No token endpoint exists; do not build
against one. When it ships (target: v1.x, additive), static API keys will
remain supported in parallel and any wind-down will follow the deprecation
policy in `overview.md`.
