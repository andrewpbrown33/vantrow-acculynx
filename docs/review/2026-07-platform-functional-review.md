# Eaverow Platform — Functional & Adversarial Review (2026-07-18)

**Scope:** full functional + backend + funnel review of `apps/platform`, driven by a generated
adversarial mock dataset loaded into demo/file-store mode. **This is a findings report — no code
was fixed.** Andrew greenlights fixes after review.

**Method:** three read-only exploration passes over every route, the data/auth/store layer, and
the marketing→Vantrow surface; a **1,504-job / 1,204-contact / 3.3 MB mock dataset** generated by
`apps/platform/scripts/mock-seed.mjs` and loaded via the file store; direct code verification of
each finding (file:line); and inspection of the rendered app. The mock dataset embeds deliberate
edge cases (an int4-overflow estimate, a multi-signed job, a paid job with a fresh draft, an
oversized signature, plus duplicate/near-duplicate/name-only contacts).

**Reproduce the harness:** `node apps/platform/scripts/mock-seed.mjs` writes `.data/platform.json`
(gitignored); then run the platform in demo mode (no Supabase env). The script is committed; the
dataset is not.

---

## Executive summary

The core loop (lead → estimate → e-sign → invoice → collect) is **functionally sound and well
built** — money is integer-cents, inputs are largely validated, org isolation is enforced in two
layers, and the happy path works end to end. The problems cluster in three areas:

1. **A confirmed multi-tenant security hole** (RLS lets a user join any org as owner) — the one
   item that must be fixed before real customers touch Supabase mode.
2. **Scale**: the pipeline — the app's *default landing page* — does ~1,505 sequential full-file
   reads (file mode) / ~7,500 queries (Supabase mode) per render, and **failed to render within
   2 minutes** on 1,504 jobs. A real switching contractor's data will make the home screen unusable.
3. **The funnel**: SIGN UP is well-guided, but **MIGRATE → POPULATE PIPELINE breaks** — importing
   contacts creates no jobs, the pipeline empty-state is a dead end, and the marketing site has no
   link to the platform at all.

Nothing here is a rewrite; the fixes are targeted. Severity table below, details after.

## Details

### 1. [CRITICAL] RLS privilege escalation via self-inserted membership
`0003_platform_auth_rls.sql:55-57` — the `memberships` INSERT policy's `WITH CHECK` is only
`user_id = auth.uid()`. It does **not** constrain `org_id` or `role`. Since every other tenant
table's policy grants access via `org_id in (select org_id from memberships where user_id =
auth.uid())`, an authenticated user can `insert into memberships (user_id, org_id, role) values
(self, <any org id>, 'owner')` and gain full read/write to that tenant.
**Repro:** any authed Supabase session → insert a membership row for a known/guessed org UUID →
that org's jobs/contacts/invoices become readable/writable.
**Why it exists:** org bootstrap runs via the **service role** (which bypasses RLS), so an
authenticated-client INSERT policy on memberships is never actually needed by the app.
**Mitigation today:** only org UUID unguessability (not a control).
**Recommended fix:** drop the `members insert own memberships` policy entirely (bootstrap doesn't
use it); if/when team-invites ship, gate membership creation behind a server-side invite check,
never a self-insert. Also add UPDATE/DELETE policies (currently none) so role can't be self-escalated.

### 2. [HIGH] Pipeline render is pathological at scale
The pipeline (`/` redirects here) loads every job's full bundle. `pipeline/page.tsx:30` calls
`store.getJobBundle(job.id)` for **every** job. In file mode, `read()` (`store.ts:162`) re-reads
and `JSON.parse`s the **entire dataset file on every call** (`readOrSeed`, `store.ts:147-149`),
serialized through one queue; `getJobBundle` is one such read (`store.ts:386`). So the page does
`1 (jobsByStage) + N (getJobBundle)` = **~1,505 sequential 3.3 MB parses** for 1,504 jobs. In
Supabase mode, `getJobBundle` (`store.ts:1028`) fires **5 queries per job** → ~1 + 5N ≈ **7,500
queries** per render, and the route is `force-dynamic` so it happens on every hit.
**Repro:** load the mock dataset, open `/pipeline` — **it did not finish rendering within 120 s**
in this review.
**Impact:** the app's home screen is unusable for exactly the switching contractor we target
(hundreds–thousands of jobs).
**Recommended fix:** compute pipeline cards from a single pass — `jobsByStage()` already returns
the jobs; join contacts + a per-job value in one read (file mode) / one or two batched queries
(Supabase: fetch contacts and the latest estimate per job set with `in (...)`), and drop the
per-job `getJobBundle`. Add pagination or per-column limits. Consider caching / removing
`force-dynamic` where possible.

### 3. [HIGH] Money overflow — no upper bound on line values
`normalizeOptions` (`actions.ts:69-83`) enforces finite / `qty>0` / `price>=0` but sets **no upper
bound**. `optionSubtotalCents` (`money.ts:9`) can exceed the Postgres `int4` cap (~2.147e9 cents =
$21.4M) that `invoices.subtotal_cents/total_cents` use (`0002_platform.sql:118-120`).
**Repro (in the mock data):** the "Overflow Testcase LLC" job carries an invoice with
`totalCents = 7,000,000,000` ($70M). File mode stores it fine; Supabase mode would throw `22003
numeric_value_out_of_range` at `createInvoice` (`actions.ts:308`) — an **unhandled** store throw.
Also `qty * unitPriceCents` beyond 2^53 loses integer precision in JS before it ever reaches the DB.
**Impact:** a legitimately large commercial estimate 500s on invoice creation in prod; dev never
reproduces it (file/prod divergence).
**Recommended fix:** clamp/validate line values to a sane max (e.g. reject `unitPriceCents >
100_000_00` and `quantity > 100_000`, cap subtotal), OR widen the money columns to `bigint` and
guard against 2^53. Wrap store writes so DB constraint violations surface as user errors, not throws.

### 4. [HIGH] Signup double-bootstrap race → duplicate orgs
Org bootstrap runs in two places: right after signup (`auth-actions.ts:142`) and again on first
authenticated load (`session.ts:112`). Each guards on "does a membership exist" but the
check→insert isn't atomic, and each creates a **new** `gen_random_uuid()` org — so
`unique(user_id, org_id)` (`0003:41`) can't dedupe (the org_ids differ). A user who signs up and
immediately loads can get **two orgs/memberships**; `resolveSupabaseSession` then picks one
arbitrarily (`session.ts:76`), as does `signIn`'s fresh-workspace check.
**Recommended fix:** bootstrap in exactly one place, or make it atomic — e.g. an `on conflict`
upsert keyed by `user_id`, or a "one org per owner" guard (`select ... where user_id` before
insert inside a transaction / RPC). Prefer a DB function so the check+insert is atomic.

### 5. [HIGH] Public e-sign path is unbounded and non-expiring
`/sign/[token]` runs entirely as **service-role** (`getServiceStore`, `actions.ts:237`), scoped
only by `send_token`. Issues: (a) the token **never expires** despite user-facing "invalid or has
expired" copy (`actions.ts:241`) — it's reusable until the estimate flips to `signed`; (b)
`imageDataUrl` is validated only by `startsWith("data:image")` (`actions.ts:257`) and stored as
`text` (`0002:101`) with **no size cap and no rate limit** — a multi-MB or non-image payload can
be POSTed through the RLS-bypassing path.
**Repro (mock data):** the "Oversized Signature Blob" job carries a 260 KB `imageDataUrl`; nothing
rejects far larger.
**Recommended fix:** add a `send_token` expiry (e.g. `sent_at + 30 days`) and check it; cap
`imageDataUrl` length (e.g. ≤ 200 KB) and validate it's a real PNG/JPEG data URL; add a simple
per-token / per-IP rate limit on the sign endpoint (the marketing waitlist already has a rate-limit
helper pattern to mirror).

### 9. [MED] Invoice-number race
`createInvoice` computes `number = INV-${1001 + countInvoices(org)}` (`actions.ts:309`) —
non-atomic read-then-insert. Two concurrent invoice creations for one org compute the same number
→ violates `unique(org_id, number)` (Supabase 500) or silently duplicates (file mode).
**Fix:** a per-org sequence / counter in a transaction, or an `on conflict` retry.

### 10. [MED] Unguarded backward state transitions
`updateEstimate` (`store.ts:302`) has **no status guard** — a `signed` estimate's
`options/taxRatePct/discountCents` can still be patched, changing `estimateTotals` after an invoice
was frozen from the old totals → invoice/estimate mismatch. `createEstimate` (`actions.ts:163`) and
`signEstimate` (`actions.ts:276`) unconditionally set `estimating`/`won` even on an already
`invoiced`/`paid` job. The mock data's "Repeat Customer — Backward Transition" job (paid, with a
fresh draft estimate) and "Multi-Signed Household" (two signed estimates) exercise this.
**Fix:** guard transitions — reject estimate edits once `status='signed'`; forbid creating/signing
estimates on jobs already `invoiced`/`paid` (or branch to a "change order" concept); make
`createInvoice` pick the **latest** signed estimate deterministically.

### 11. [MED] Payments against void invoices
`recordPayment` (`actions.ts:352`) checks only `balance <= 0`, never `status === 'void'`. A void
invoice with a positive balance still accepts payments and flips to `paid`.
**Fix:** reject payment when `status !== 'open'`.

### 12. [MED] Import dedupe false-positives
`importContacts` (`migration-actions.ts:65`) dedupes by lowercased email **OR** digits-only phone.
A genuinely new contact that happens to share a phone (family, shared office line) with an existing
one is dropped as a duplicate (silent data loss). Name-only rows (no email/phone) are never
deduped, so re-running an import multiplies them.
**Repro (mock data):** 80 exact-dup, 70 same-phone/different-email, and 50 name-only contacts.
**Fix:** dedupe on (email) primarily, treat phone as a weaker signal (match only when name also
matches), and dedupe name-only rows by normalized name within the import batch. Show the user what
was merged and let them undo.

### 13. [MED] False "you've been notified" claim on signing
`sign-form.tsx:90-92` tells the homeowner: *"Your signature has been recorded and {org} has been
notified. A copy of your approved … option will be on its way."* But `signEstimate`
(`actions.ts:231-288`) only writes an in-app `activity` row — there is **no email/notification code
anywhere** in the platform. The contractor is not notified and no copy is sent.
**Fix:** either soften the copy to what's true ("Your signature is recorded — {org} will see it in
their dashboard"), or (better) wire a real notification (email the contractor + a PDF copy to the
homeowner). This overlaps the missing transactional-email work.

### 14. [MED, funnel] Onboarding's strongest CTA leaves the app
On `/onboarding` the only solid-brand button is the concierge **`mailto:`** (`onboarding/page.tsx:209`)
— it opens an email client and leaves the product. The one strong in-app "continue" ("View
contacts") appears **only after** a successful CSV import (`contact-importer.tsx`). A user who picks
the concierge path, or who reads and scrolls, gets **no in-app next step** except a muted "Skip for
now → dashboard." **Fix:** add a persistent primary "Continue → your pipeline / import contacts"
CTA, and give the concierge path an in-app acknowledgement + next step.

## Funnel analysis — SIGN UP → MIGRATE → POPULATE PIPELINE

**SIGN UP — solid.** signup → confirm → onboarding, with a fresh-workspace redirect safety net
(`auth-actions.ts:203`) and the confirmation-link resilience fixed earlier. Minor: see #4 race.

**MIGRATE — content-rich but leaky.** The onboarding page is genuinely good (export playbook +
integration harvesting + CSV importer + concierge). But its forward momentum depends entirely on
the CSV importer's success card; every other path (read the export steps, use integration
harvesting, choose concierge) ends with **no in-app next step** (#14). And the concierge button —
the most prominent — leaves the app.

**POPULATE PIPELINE — this is where the funnel breaks.** Three compounding gaps:
- **#7 No bulk bridge.** `importContacts` deliberately creates no jobs (`migration-actions.ts:33`).
  After importing 1,000 contacts, the board is still empty; the only way to populate it is clicking
  **"Start a job" one contact at a time** (`contacts/page.tsx:118`), which also has no dedupe (each
  click spawns another lead).
- **#6 The pipeline empty-state is a dead end.** `/` redirects to `/pipeline`, which renders six
  empty "No jobs" columns with only a small "New lead" button (`pipeline/page.tsx:91`) — **no hero,
  no "Import from AccuLynx," no nudge.** Contrast `/contacts`, which has a proper empty state. The
  app's default landing page does nothing to sell or trigger migration.
- **#15 Re-login reinforces it.** After contacts are imported (but no jobs), later logins route to
  `/pipeline` (barren), not back to migrate (`auth-actions.ts:203`).

**#8 The biggest funnel gap of all is upstream:** the **marketing site has no link to the
platform**. Every site CTA points to `/early-access` (a waitlist that stores a lead), and the real
org-creating `/signup` is **orphaned** — reachable only if you already know the URL. So "get early
access" and "sign up for the product" are two disconnected universes with a manual hand-off. This
is a Vantrow/funnel decision (covered in the design doc) but it's the top-of-funnel leak.

### The 5 highest-leverage funnel repairs (recommended, for greenlight)
1. **Pipeline empty-state** → a real first-run hero: "Bring your AccuLynx jobs over" + primary
   "Import from AccuLynx" (→ onboarding importer) + "Add your first lead." (Fixes #6, the default
   landing page.)
2. **Bulk contacts → pipeline** → a "Select contacts → Start jobs" batch action on `/contacts`,
   with dedupe (don't spawn a second job for a contact that already has one). (Fixes #7.)
3. **Wire marketing → platform** → point the site's primary CTA (or a "Log in / Start free" header
   link) at the platform `/signup`, and/or auto-provision from the waitlist. (Fixes #8.)
4. **Onboarding "Continue" CTA** → a persistent primary next-step, and an in-app acknowledgement for
   the concierge path. (Fixes #14.)
5. **Fix the pipeline perf (#2)** so the populated board is actually usable — otherwise "populate
   pipeline" succeeds into an unusably slow screen.

## What's genuinely good (keep)
- Money is integer-cents end to end; `estimateTotals` clamps discount and rounds tax sensibly.
- Input validation on estimates/payments/imports is mostly thorough (finite checks, tier presence,
  no-overpay, honeypot/rate-limit on the public forms).
- Two-layer org isolation (app-level `orgId` checks + RLS) — modulo finding #1.
- The confirmation-link resilience and error-surfacing added earlier hold up.
- The onboarding content (export playbook, integration harvesting, importer) is strong material.
